# 数据质量测试为什么只能做"部分"

## 问题核心

数据质量测试需要**给任务打分**，但打分需要**LLM判断**。

---

## 完整的数据质量测试流程

```
┌─────────────────────────────────────────────────────┐
│  步骤1: 读取任务                                      │
│  {"id": "task_1", "ticket": "能吃硝苯地平吗？"}     │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  步骤2: LLM评分（❌ 需要API，这里卡住了）              │
│                                                      │
│  评估维度：                                           │
│  • 临床准确性 (0-10分) - 需要LLM判断医学内容是否正确   │
│  • 场景真实性 (0-8分)  - 需要LLM判断场景是否真实      │
│  • 评估完整性 (0-7分)  - 需要LLM判断评估标准是否完整   │
│  • 难度适当性 (0-5分)  - 需要LLM判断难度是否合适       │
│                                                      │
│  需要调用:                                           │
│  OpenAI API: "请评估这个医学任务的质量..."           │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  步骤3: 得到分数（没有API就得不到真实分数）            │
│  {"clinical_accuracy": 8.5, "total": 27.3}         │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  步骤4: 质量分级（✅ 这部分能运行）                    │
│  • ≥27分 → 高质量（保留）                            │
│  • 24-26分 → 中等质量（改进）                        │
│  • <24分 → 低质量（剔除）                            │
└─────────────────────────────────────────────────────┘
```

## 当前测试做了什么

### ✅ 能做的（不需要API）

```bash
# 1. 数据格式验证
python -m DataQualityFiltering data-quality check --input tasks.json

输出: ✓ 数据格式正确
      ✓ 包含必需字段
      ✓ JSON格式有效
```

```bash
# 2. 质量阈值筛选（使用默认分数）
python -m DataQualityFiltering data-quality quality-threshold \
  --input tasks.json --no-improvement

输出: ⚠️ 未提供分数列表，将使用默认分数
      平均分数: 22.0/30 (默认值，不是真实分数!)
```

### ❌ 不能做的（需要API）

```bash
# 3. 实际质量评分（需要LLM）
# ❌ 没有API Key无法运行

需要评估每个任务的:
- "这个医学问题是否准确？" → 需要LLM判断
- "这个场景是否真实？" → 需要LLM判断
- "评估标准是否完整？" → 需要LLM判断
```

## 为什么不能绕过LLM？

### 问题: 能否用规则代替LLM评分？

**不能**，原因：

1. **临床准确性需要医学知识**
   ```
   规则: "是否有禁忌症?"
   → 无法判断内容是否正确
   → LLM需要知道"硝苯地平"的禁忌症是什么
   ```

2. **场景真实性需要常识判断**
   ```
   规则: "场景是否真实?"
   → 需要理解患者描述是否合理
   → "血压有点高，能不能吃硝苯地平" 是真实场景
   → "我血压300，能吃10片硝苯地平吗" 是不真实场景
   ```

3. **评估完整性需要理解能力**
   ```
   规则: "是否评估了安全性?"
   → 需要理解evaluation_criteria的内容
   → 需要判断是否覆盖了必要的安全检查点
   ```

## 数据质量测试的能力边界

| 测试项 | 不需要API | 需要API |
|-------|----------|---------|
| **格式验证** | ✅ JSON格式、字段完整性 | - |
| **场景分类** | ✅ 关键词匹配 | - |
| **规则生成** | ✅ 基于配置生成 | - |
| **质量评分** | ❌ 只能用默认分 | ✅ LLM评估内容质量 |
| **质量分级** | ✅ 分级逻辑正确 | - |
| **真实质量** | ❌ 无法判断 | ✅ 需要真实分数 |

---

## 总结

**"部分"指的是**:
- ✅ 验证了**流程**能运行（格式检查→分类→分级）
- ❌ 无法获得**真实的质量分数**（需要LLM）

**比喻**:
- 就像考试阅卷系统
- ✅ 我们验证了"系统能接收试卷、能按分数分级"
- ❌ 但没有"老师"来给试卷打分（LLM就是老师）

**要完成完整测试，需要**:
1. LLM API Key
2. 让LLM评估每个任务的质量
3. 使用真实分数进行分级
